# 곳간 활용·연계 설계 노트 v0.1

> "찾는 것"이 아니라 "쓰는 것"이 종착점이다. 개방데이터의 실사용은 ①분석 반입 ②시스템 연계 두 갈래이고,
> 곳간의 도구는 각 갈래의 다음 행동까지 안내해야 한다.

## 1. 소비 시나리오 3종과 연계 방식 판정

| 시나리오 | 전형 사례 (지자체) | 권장 연계 방식 | 판정 근거 필드 |
|---|---|---|---|
| 일회성 분석 | 농작물 재배면적 × 직불금 지급내역 교차검증 | 파일데이터 직접 다운로드 or 자동변환 API 1회 호출 | 제공형태, 매체유형 |
| 주기 배치 | 감사·점검용 계약현황 월간 적재 | 파일 배치 반입(폐쇄망) or 자동변환 API 스케줄 | 업데이트 주기, 차기 등록 예정일 |
| 실시간 연계 | 민원 알림·대시보드 | Open API 운영계정 (심의 소요 감안) | API 유형, 신청가능 트래픽, 심의 유형 |

핵심 판정 로직: **갱신주기 × 망환경 × 트래픽** 세 축이면 대부분 결정된다.
- 데이터 갱신이 "연 1회"인데 실시간 API 연계를 설계하는 것은 과잉 — 파일 배치가 정답.
- 폐쇄망 내부 시스템은 API 실시간 연계 불가 → 망연계 구간(DMZ 중계) 또는 반입 배치로 강제 강등.
- 개발계정 트래픽(예: 일 500건)으로 운영 서비스를 돌리는 설계 오류를 사전 차단.

## 2. 분석 반입 관점 — v0.2 도구 스펙

### `preview_file_data`  [온라인 레인]
파일데이터 다운로드 URL에서 선두 N행을 받아 인코딩(UTF-8/CP949)·구분자·칼럼명·추정 타입·표본값을 반환.
분석가가 "받아보니 못 쓰는 데이터"를 다운로드 전에 거른다.
- 반환: `{encoding, delimiter, columns:[{name, dtype_guess, sample}], row_estimate, status}`
- 정직성: 로그인 필요/장애 파일은 `fetch_error`로, "데이터 품질 나쁨"으로 번역 금지.

### `find_joinable`  [오프라인 레인]
주어진 데이터셋과 결합 가능한 후보를 찾는다. 결합키 우선순위:
1. **행정표준코드** — 법정동코드·행정기관코드가 칼럼에 있으면 최상급 조인키
2. **표준데이터** — 목록유형=표준(공공데이터 개방 표준)은 스키마가 고정이라 기관 간 수평 결합 가능
3. 분류체계·제공기관 동일 계열 (약한 후보 — low_confidence로 표기)
- 주의: 칼럼 정보는 첨부/미리보기 파싱 전까지 목록 메타데이터 수준 추정 → 확신도 명시 필수.

## 3. 시스템 연계 관점 — v0.2 도구 스펙

### `assess_linkage`  [오프라인 레인, 판정 엔진]
입력: 데이터셋 `_id` + 사용 시나리오(일회성/배치/실시간) + 망환경(인터넷망/폐쇄망)
출력: 연계 방식 판정 + 절차 체크리스트 + 리드타임 경고
```
예) 시나리오=실시간, 망=폐쇄망, 심의유형=심의승인, 트래픽=개발 500
→ 판정: "직접 실시간 연계 불가. ①망연계 중계 설계 필요 ②운영계정 심의 리드타임 산정
   ③갱신주기(월 1회)를 볼 때 실시간 요건 자체를 재검토 권고(배치로 충분)"
```
판정문은 근거 필드를 반드시 인용 — 근거 없는 권고는 생성하지 않는다.

### 제공기관 관점 (자기점검 모드)
곳간 인덱스에는 **자기 기관이 제공기관인 목록**도 들어있다. 활용자이면서 제공자인 공공기관 특성상:
- `search_dataset(org="<기관명>")` + 차기 등록 예정일 필터 → **갱신기한 도래/경과 목록 자동 점검**
- 공공데이터 제공 운영실태 평가 대비 상시 모니터링에 그대로 쓰인다.
- v0.2: 스냅샷 간 diff로 "자기 기관 목록 중 이번 달 갱신 누락" 알림.

## 4. 법령검토 MCP 크로스레인

| 곳간 출력 | 법령검토 MCP 입력 | 연계 질의 예 |
|---|---|---|
| 보유근거 (법령명) | 법령 조회 | "이 데이터의 보유근거 법령상 제3자 제공 제한이 있는가" |
| 데이터 한계·이용허락범위 | 계약검토 레인 | "이 API를 용역 산출물에 포함해도 되는가" |
| 심의유형·트래픽 | 감사 레인 | "사업계획서의 실시간 연계 설계가 승인유형과 정합한가" |

마지막 행이 실전 가치가 크다 — **용역 제안서의 데이터 연계 설계를 곳간으로 사실검증**하는 감사 시나리오.
(제안서: "○○ API 실시간 연동" → 곳간: 해당 API는 심의승인·개발 500건 → 지적사항 도출)

## 5. 벤치마크 반영 (H레인 증설)

- H3. 시나리오·망환경을 주고 연계 방식 판정 + 근거 필드 인용을 채점 *(근거 없는 권고 = 게이트)*
- H4. 용역 제안서 발췌(API 연계 설계 부분)를 주고 실현 가능성 검증 — 트래픽·심의·갱신주기 3축 지적
- H5. 지정 기관 제공 목록 중 차기 등록 예정일 경과 건 식별 (자기점검 모드)
